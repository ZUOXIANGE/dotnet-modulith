# S3 预签名上传实践（DotNetModulith）

本文档描述 DotNetModulith 项目中文件上传的落地实现，基于 `Storage` 模块与业务模块（`Books`、`Users`）的协作模式。所有内容均以当前代码库为依据，并在末尾标注已实现能力与未来演进方向。

---

## 一、设计结论

项目采用 **“后端签发上传会话 + 前端直传对象存储 + 业务接口内验收绑定”** 的方案，而不是 “所有文件先传 `temp/` 再移动到 `formal/`”。

核心要点：

1. `Storage` 模块统一负责签发上传会话、生成预签名 URL、验收对象、签发下载 URL
2. 客户端拿到预签名 URL 后直接 PUT 到 RustFS（S3 兼容），应用服务器不承载文件流量
3. 默认直接上传到最终对象 Key，不做额外 “移动”
4. 业务模块（如 `Books`、`Users`）在自身命令处理流程内调用 `IStorageUploadSessionService.ConsumeUploadSessionAsync` 完成验收，前端无需额外调用 “完成上传” 接口
5. 对象 Key 由后端生成，不信任前端文件名
6. 预签名 URL 默认 15 分钟过期，桶默认通过 RustFS 私有访问

一句话总结：**业务成功 = 文件已上传 + 业务接口内验收通过 + 事务提交成功**。

## 四、标准上传流程

### 4.1 签发上传会话

**接口**：`POST /api/storage/upload/presign`（需 `storage.upload` 权限）

请求体 [CreatePresignedUploadRequest.cs](file:///d:/Code/dotnet-modulith/src/DotNetModulith.Modules.Storage/Api/Contracts/Requests/CreatePresignedUploadRequest.cs)：

```json
{
  "fileName": "cover.jpg",
  "contentType": "image/jpeg",
  "purpose": "book-cover"
}
```

响应 [PresignedUploadResponse.cs](file:///d:/Code/dotnet-modulith/src/DotNetModulith.Modules.Storage/Api/Contracts/Responses/PresignedUploadResponse.cs)：

```json
{
  "uploadId": "01JX....",
  "objectKey": "books/covers/2026/06/22/6f5f2d3c....jpg",
  "uploadUrl": "http://localhost:9000/modulith-files/...",
  "expiresAtUtc": "2026-06-22T10:30:00Z"
}
```

**服务端动作**（见 `CreateUploadSessionAsync`）：

1. 校验 `purpose` 是否受支持
2. 校验声明的 `contentType` 是否在允许范围内（当前仅限图片）
3. 生成服务端控制的对象 Key
4. 创建 `UploadSessionEntity`（状态 `Pending`）
5. 调用 `IObjectStorageService.CreatePresignedUploadAsync` 生成预签名 PUT URL
6. 落库并返回会话描述符

### 4.2 对象 Key 生成策略

见 `BuildPurposeObjectKey`：

```text
{folder}/{yyyy/MM/dd}/{guid:N}{ext}
```

| Purpose       | Folder          |
| ------------- | --------------- |
| `book-cover`  | `books/covers`  |
| `user-avatar` | `users/avatars` |
| 其他          | `uploads`       |

关键原则：

- 使用 GUID 保证唯一性，不使用用户原始文件名
- 按业务用途分目录，便于生命周期管理与审计
- 原始文件名仅存于 `UploadSessionEntity.OriginalFileName`，不参与存储路径

### 4.3 客户端直传 RustFS

前端拿到 `uploadUrl` 后直接 PUT 文件二进制：

```typescript
// frontend/src/utils/api.ts
export async function uploadToPresignedUrl(uploadUrl: string, file: File): Promise<void> {
  const response = await fetch(uploadUrl, {
    method: 'PUT',
    body: file,
    headers: { 'Content-Type': file.type || 'application/octet-stream' }
  })
  if (!response.ok) throw new Error(`上传失败: ${response.status}`)
}
```

约束：

- 使用 `PUT` 直接上传原始二进制，不使用 `FormData`
- `Content-Type` 必须与签发时声明的一致
- 上传成功后前端仅持有 `uploadId`，不认为业务已完成

### 4.4 业务接口内验收绑定

**这是当前实现的关键设计**：业务模块在自身命令处理流程内调用 `ConsumeUploadSessionAsync` 完成验收，前端无需额外调用 “完成上传” 接口。

以 `Books` 模块为例，见 [BookService.cs](file:///d:/Code/dotnet-modulith/src/DotNetModulith.Modules.Books/Application/BookService.cs)：

```csharp
public async Task<BookDetails> CreateBookAsync(CreateBookInput input, CancellationToken ct)
{
    // ... 业务校验 ...

    var coverImageUrl = string.Empty;
    if (input.CoverUploadId.HasValue)
    {
        var cover = await _storageUploadSessionService.ConsumeUploadSessionAsync(
            input.OperatorUserId,
            input.CoverUploadId.Value,
            UploadPurposes.BookCover,
            ct);
        coverImageUrl = cover.ObjectUrl;
    }

    var entity = BookEntity.Create(..., coverImageUrl, now);
    _dbContext.Books.Add(entity);
    await _dbContext.SaveChangesAsync(ct);
    return ...;
}
```

`ConsumeUploadSessionAsync` 内部执行（见 [StorageUploadSessionService.cs](file:///d:/Code/dotnet-modulith/src/DotNetModulith.Modules.Storage/Application/StorageUploadSessionService.cs)）：

1. 校验会话是否存在
2. 校验 `OwnerUserId` 是否属于当前用户
3. 校验 `purpose` 是否与业务场景匹配
4. 校验会话是否已被消费（幂等保证）
5. 校验会话是否过期（过期则标记 `Expired` 并拒绝）
6. 调用 `GetObjectInfoAsync`（`HeadObject`）确认对象真实存在
7. 校验对象大小 `> 0`
8. 校验对象实际 `ContentType` 仍在允许范围内
9. 标记会话为 `Consumed`
10. 返回 `ConsumedStorageObject`（含 `ObjectUrl`、`ContentType`、`Size`）

**重要原则**：浏览器上传返回 200 ≠ 业务成功。只有业务接口事务提交成功才算完成。

### 4.5 前端交互流程

以图书封面为例（见 [CoverUpload.vue](file:///d:/Code/dotnet-modulith/frontend/src/components/CoverUpload.vue)）：

1. 用户选择文件后，前端做本地大小（10MB）与扩展名（jpg/png/webp）预校验
2. 调用 `createUploadSession(fileName, contentType, 'book-cover')` 获取预签名 URL
3. 调用 `uploadToPresignedUrl(uploadUrl, file)` 直传 RustFS
4. 前端保存 `uploadId`，使用 `URL.createObjectURL(file)` 生成本地预览
5. 用户继续填写图书表单
6. 用户点击 “提交”，前端只调用一次 `POST /api/books`，请求体包含 `coverUploadId`
7. 以业务接口响应作为最终成功依据

## 六、内容类型与大小校验

当前实现采用双层校验：

### 6.1 签发时校验（第一层）

`ValidateContentType` 在 `CreateUploadSessionAsync` 中执行：

```csharp
private static readonly HashSet<string> ImageContentTypes =
[
    "image/jpeg", "image/png", "image/webp", "image/gif"
];
```

- 仅允许图片类型
- `purpose` 必须是 `book-cover` 或 `user-avatar`

### 6.2 验收时校验（第二层）

`ConsumeUploadSessionAsync` 通过 `GetObjectInfoAsync` 获取对象实际 `ContentType`，再次校验：

- 对象必须存在（`HeadObject` 成功）
- 对象大小 `> 0`
- 实际 `ContentType` 必须在允许范围内

### 6.3 大小限制

- 直传接口 `POST /api/storage/upload/direct` 通过 `[RequestSizeLimit(104857600)]` 限制 100MB
- 预签名 PUT 的严格大小控制依赖：前端预校验 + 验收时 `GetObjectInfoAsync` 返回的实际 `Size`
- 当前未在 `StorageOptions` 中按 `purpose` 配置差异化大小上限，属未来演进项

## 七、安全实践

### 7.1 权限控制

[PermissionCodes.cs](file:///d:/Code/dotnet-modulith/src/DotNetModulith.Abstractions/Authorization/PermissionCodes.cs) 定义：

- `storage.view` - 读取对象
- `storage.upload` - 签发上传会话与直传
- `storage.delete` - 删除对象（预留）

`StorageController` 通过 `[Authorize(Policy = ...)]` 强制校验。

### 7.2 会话归属校验

`ConsumeUploadSessionAsync` 校验 `session.OwnerUserId == ownerUserId`，防止用户消费他人的上传会话。

### 7.3 预签名 URL 时效

`PresignedUrlExpireSeconds` 默认 900 秒，会话过期后自动标记 `Expired` 并拒绝消费。

### 7.4 桶自动创建

`EnsureBucketExistsAsync` 在首次访问时自动创建桶，使用 `SemaphoreSlim` 双重检查锁保证线程安全。

### 10.2 未来演进项

以下能力当前未实现，按优先级排列，供后续迭代参考：

#### 优先级高

1. **Multipart Upload 支持**
   - 当前仅支持单次 PUT，大文件场景需补充分片上传
   - 需新增：创建分片会话、为每 Part 签发 URL、前端回传 ETag 列表、CompleteMultipartUpload

2. **按 purpose 配置大小上限**
   - 在 `StorageOptions` 或独立配置中按用途设置 `MaxSizeBytes`
   - 验收时通过 `GetObjectInfoAsync` 返回的 `Size` 强制校验

3. **替换流程的旧对象回收**
   - 当前 `UpdateBookAsync` 替换封面时，旧对象 URL 被覆盖但未删除
   - 建议引入 “新对象上线，旧对象延迟回收” 机制

#### 优先级中

4. **文件表拆分**
   - 拆分为 `FileObjects`（物理对象生命周期）+ `FileBindings`（业务绑定）
   - 优势：一个对象可被多业务引用、物理删除时易判断引用计数、支持版本与回收站

5. **异步后处理（CAP 事件）**
   - 上传完成后发布集成事件，驱动缩略图生成、CDN 刷新、审计等
   - 在业务事务内写入 CAP Outbox，由订阅者异步消费，保证最终一致性
   - 详见 [module-communication.md](file:///d:/Code/dotnet-modulith/docs/module-communication.md)

6. **软删除 + 异步物理清理**
   - 业务删除时仅取消绑定，物理对象进入延迟删除队列
   - 后台任务扫描安全窗口期后执行 `DeleteObject`

#### 优先级低

7. **隔离前缀（`quarantine/`）**
   - 仅在需要病毒扫描、内容审核、人工审批时启用
   - 当前业务（头像、封面）无需此流程

8. **生命周期规则**
   - 配置 RustFS 桶规则自动清理未完成的 Multipart Upload 与过期临时对象

9. **可观测性**
   - 采集上传凭证申请次数、上传完成成功率、验收失败率等指标
   - 结合 OpenTelemetry 插桩，详见 `aspire-service-defaults` 技能

10. **内容安全增强**
    - 当前仅校验 `ContentType`，高风险场景应增加图片实际解码验证、文档格式探测
