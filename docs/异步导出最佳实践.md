# 异步导出最佳实践（DotNetModulith）

本文档以 **“异步导出图书列表 Excel”** 为场景，描述 DotNetModulith 项目中长耗时报表导出的落地架构与实现指南。所有内容均以当前代码库的基础设施为依据，复用已有的 CAP 事件驱动、Storage 模块、Notifications 模块与权限体系。

---

## 一、设计结论

项目采用 **“提交导出任务 + CAP 事件驱动异步生成 + Storage 模块托管产物 + Notifications 模块通知用户 + 预签名 URL 下载”** 的方案，而不是 “同步接口直接返回文件流”。

核心要点：

1. 导出请求只创建一条 `export_tasks` 记录并发布 CAP 事件，HTTP 接口立即返回任务 ID
2. 订阅者在后台消费事件，流式查询数据并生成 Excel，避免阻塞 HTTP 请求
3. 生成的 Excel 通过 `Storage` 模块上传到 RustFS，业务表只保存对象 Key
4. 导出完成后通过 `Notifications` 模块向用户发送站内通知，附带下载凭证
5. 前端通过预签名 GET URL 下载文件，URL 由 `Storage` 模块动态签发
6. 复用已有的 `reports.export` 权限码

一句话总结：**导出请求轻量化、生成过程异步化、产物存储对象化、下载访问签名化**。

## 二、为什么不能同步导出

同步导出（即 `GET /api/books/export` 直接返回 `File(...)` 流）在以下场景会出问题：

| 问题     | 说明                                                                     |
| -------- | ------------------------------------------------------------------------ |
| 请求超时 | 图书数量较大时，查询 + 序列化 + Excel 生成可能超过网关或浏览器的超时阈值 |
| 内存压力 | 一次性加载全量数据到内存生成 Excel，容易触发 OOM                         |
| 不可重试 | 网络抖动导致失败后，用户只能从头再来，无法断点续传                       |
| 无法观测 | 导出失败没有记录，运维难以排查                                           |
| 资源竞争 | 长耗时请求占用 Web 服务器线程池，影响其他接口响应                        |
| 无法通知 | 用户离开页面后无法得知导出是否完成                                       |

异步导出将 “请求” 与 “执行” 解耦，请求侧只做轻量记录，执行侧在后台按节奏消费，是生产环境的推荐做法。

```csharp
public sealed class ExportTaskEntity
{
    public Guid Id { get; private set; }
    public Guid RequestedUserId { get; private set; }
    public string ExportType { get; private set; } = string.Empty;
    public string FilterPayload { get; private set; } = string.Empty;
    public ExportTaskStatus Status { get; private set; }
    public int EstimatedRows { get; private set; }
    public int ProcessedRows { get; private set; }
    public string? ObjectKey { get; private set; }
    public long? SizeBytes { get; private set; }
    public int TotalRows { get; private set; }
    public string? LastError { get; private set; }
    public DateTimeOffset CreatedAt { get; private set; }
    public DateTimeOffset UpdatedAt { get; private set; }
    public DateTimeOffset? CompletedAt { get; private set; }
    public DateTimeOffset? ExpiresAt { get; private set; }

    private ExportTaskEntity() { }

    public static ExportTaskEntity Create(
        Guid requestedUserId,
        string exportType,
        string filterPayload,
        int estimatedRows,
        DateTimeOffset now)
        => new()
        {
            Id = Guid.NewGuid(),
            RequestedUserId = requestedUserId,
            ExportType = exportType,
            FilterPayload = filterPayload,
            Status = ExportTaskStatus.Pending,
            EstimatedRows = estimatedRows,
            CreatedAt = now,
            UpdatedAt = now
        };

    public void MarkRunning(DateTimeOffset now)
    {
        Status = ExportTaskStatus.Running;
        UpdatedAt = now;
    }

    public void UpdateProgress(int processedRows, DateTimeOffset now)
    {
        ProcessedRows = processedRows;
        UpdatedAt = now;
    }

    public void MarkSucceeded(string objectKey, long sizeBytes, int totalRows, DateTimeOffset now)
    {
        Status = ExportTaskStatus.Succeeded;
        ObjectKey = objectKey;
        SizeBytes = sizeBytes;
        TotalRows = totalRows;
        ProcessedRows = totalRows;
        CompletedAt = now;
        ExpiresAt = now.AddDays(7);
        UpdatedAt = now;
    }

    public void MarkFailed(string error, DateTimeOffset now)
    {
        Status = ExportTaskStatus.Failed;
        LastError = error;
        CompletedAt = now;
        UpdatedAt = now;
    }
}
```

新增字段说明：

- `EstimatedRows`：提交时预估的总行数，用于前端展示进度条与数据量预检
- `ProcessedRows`：已处理行数，每批次拉取后更新，支持前端轮询展示实时进度

## 五、标准导出流程

### 5.1 提交导出任务

**接口**：`POST /api/reports/books/export`（需 `reports.export` 权限）

请求体：

```json
{
  "keyword": "domain",
  "categoryId": "00000000-0000-0000-0000-000000000000"
}
```

响应：

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "exportTaskId": "01JX....",
    "status": "Pending",
    "estimatedRows": 50000,
    "createdAt": "2026-06-22T10:00:00Z"
  }
}
```

**服务端动作**（见 `ExportTaskService.CreateBookExportTaskAsync`）：

1. 校验当前用户拥有 `reports.export` 权限（由 `[Authorize]` 完成）
2. 序列化筛选条件为 `FilterPayload`（JSON）
3. **数据量预检**：调用 `IBookExportQueryService.EstimateRowsAsync` 预估导出行数，超过阈值（默认 10 万行）则拒绝并返回业务错误码
4. 创建 `ExportTaskEntity`（状态 `Pending`，写入 `EstimatedRows`）
5. 在 CAP 事务内保存任务并发布 `BookExportRequestedIntegrationEvent`
6. 立即返回任务 ID，不等待生成完成

参考项目现有的事务性发布模式（见 [BorrowingService.cs:144](file:///d:/Code/dotnet-modulith/src/DotNetModulith.Modules.Borrowing/Application/BorrowingService.cs#L144)）：

```csharp
public async Task<ExportTaskDescriptor> CreateBookExportTaskAsync(
    Guid requestedUserId, CreateBookExportInput input, CancellationToken ct)
{
    var filterPayload = JsonSerializer.Serialize(input);

    var estimatedRows = await _bookQueryService.EstimateRowsAsync(input.Keyword, input.CategoryId, ct);
    if (estimatedRows > MaxExportRows)
        throw new BusinessException(ApiCodes.Reports.ExportRowsExceeded,
            $"导出行数 {estimatedRows} 超过上限 {MaxExportRows}，请缩小筛选范围");

    var now = DateTimeOffset.UtcNow;
    var task = ExportTaskEntity.Create(
        requestedUserId, ExportTypes.BookList, filterPayload, estimatedRows, now);

    using var transaction = _dbContext.Database.BeginTransaction(_capPublisher, autoCommit: false);

    _dbContext.ExportTasks.Add(task);
    await _dbContext.SaveChangesAsync(ct);

    await _capPublisher.PublishAsync(
        nameof(BookExportRequestedIntegrationEvent),
        new BookExportRequestedIntegrationEvent(
            task.Id, requestedUserId, ExportTypes.BookList, filterPayload),
        cancellationToken: ct);

    transaction.Commit();

    return new ExportTaskDescriptor(task.Id, task.Status.ToString(), task.EstimatedRows, task.CreatedAt);
}
```

`MaxExportRows` 建议通过 Options 配置，默认 100000。

### 5.2 异步生成 Excel

订阅者 `BookExportJob` 消费 `BookExportRequestedIntegrationEvent`，放在 `Reports/Application/Subscribers/`，遵循项目订阅者约定（见 [Notifications 模块订阅者](file:///d:/Code/dotnet-modulith/src/DotNetModulith.Modules.Notifications/Application/Subscribers/)）。

```csharp
namespace DotNetModulith.Modules.Reports.Application.Subscribers;

public sealed class BookExportJob : ICapSubscribe
{
    private readonly ReportsDbContext _dbContext;
    private readonly IBookExportQueryService _bookQueryService;
    private readonly IExcelBuilder _excelBuilder;
    private readonly IObjectStorageService _objectStorageService;
    private readonly ICapPublisher _capPublisher;
    private readonly ILogger<BookExportJob> _logger;

    public BookExportJob(/* 依赖注入 */) { ... }

    [CapSubscribe(nameof(BookExportRequestedIntegrationEvent))]
    public async Task HandleAsync(BookExportRequestedEvent @event, CancellationToken ct)
    {
        var task = await _dbContext.ExportTasks
            .AsTracking()
            .SingleOrDefaultAsync(x => x.Id == @event.ExportTaskId, ct);

        if (task is null || task.Status != ExportTaskStatus.Pending)
            return;

        var now = DateTimeOffset.UtcNow;
        task.MarkRunning(now);
        await _dbContext.SaveChangesAsync(ct);

        try
        {
            var filter = JsonSerializer.Deserialize<CreateBookExportInput>(task.FilterPayload)!;

            var tempPath = Path.Combine(Path.GetTempPath(), $"export-{task.Id}.xlsx");
            int totalRows;

            await using (var fileStream = new FileStream(tempPath, FileMode.Create, FileAccess.Write, FileShare.None,
                bufferSize: 81920, useAsync: true))
            {
                totalRows = await _excelBuilder.BuildBookListExcelAsync(fileStream, filter, task.Id, ct);
            }

            var objectKey = $"reports/exports/{task.Id}.xlsx";
            await using var uploadStream = new FileStream(tempPath, FileMode.Open, FileAccess.Read, FileShare.None,
                bufferSize: 81920, useAsync: true);
            await _objectStorageService.UploadStreamAsync(objectKey, uploadStream,
                "application/vnd.openxmlformats-officedocument.spreadsheetml.sheet", ct);

            var sizeBytes = new FileInfo(tempPath).Length;

            task.MarkSucceeded(objectKey, sizeBytes, totalRows, now);
            await _dbContext.SaveChangesAsync(ct);

            await _capPublisher.PublishAsync(
                nameof(BookExportCompletedIntegrationEvent),
                new BookExportCompletedIntegrationEvent(
                    task.Id, task.RequestedUserId, task.ExportType,
                    totalRows, objectKey),
                cancellationToken: ct);
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "Book export failed. TaskId={TaskId}", task.Id);
            task.MarkFailed(ex.Message, now);
            await _dbContext.SaveChangesAsync(ct);

            await _capPublisher.PublishAsync(
                nameof(BookExportFailedIntegrationEvent),
                new BookExportFailedIntegrationEvent(
                    task.Id, task.RequestedUserId, task.ExportType, ex.Message),
                cancellationToken: ct);
        }
        finally
        {
            var cleanupPath = Path.Combine(Path.GetTempPath(), $"export-{task.Id}.xlsx");
            if (File.Exists(cleanupPath))
            {
                try { File.Delete(cleanupPath); }
                catch { /* 忽略清理失败 */ }
            }
        }
    }
}
```

关键设计点：

1. **幂等保护**：检查 `task.Status == Pending`，防止 CAP 重试导致重复执行
2. **游标分批拉取**：`IExcelBuilder` 内部使用 `where id > lastId limit batchSize` 游标分页，每批处理完成后调用 `task.UpdateProgress` 更新 `ProcessedRows`，前端可轮询展示实时进度
3. **临时文件流式写入**：Excel 直接写入临时文件流，避免大文件占用内存；上传时再以流方式读取，内存占用恒定
4. **流式上传**：`IObjectStorageService.UploadStreamAsync` 以流方式上传到 RustFS，不一次性加载到内存
5. **状态机推进**：`Pending → Running → Succeeded/Failed`，每步落库保证可观测
6. **产物清理**：无论成功失败，`finally` 块清理临时文件，避免磁盘泄漏
7. **完成事件**：发布 `BookExportCompletedIntegrationEvent`，由 Notifications 模块订阅发通知

### 6.1 推荐库：MiniExcel

项目当前未引入 Excel 库，推荐使用 **MiniExcel**：

- MIT 协议，无商业授权风险
- 轻量级，零依赖，DLL 体积仅约 300KB
- 内存占用极低，天生流式设计，适合大数据量导出
- API 极简，支持 `IEnumerable<T>` 直接写入与 `Dictionary` 列定义
- 支持 `SaveAs(Stream, IEnumerable)` 流式写入，配合游标分批可做到恒定内存
- 包名：`MiniExcel`

### 6.2 实现：MiniExcelExcelBuilder

新增于 `Reports/Application/MiniExcelExcelBuilder.cs`，标记 `internal sealed`。

核心设计对应时序图步骤 9-13：**游标分批拉取 → 数据转换 → 流式追加写入临时文件 → 更新进度 → 关闭流**。MiniExcel 的 `SaveAs(Stream, IEnumerable)` 重载支持流式写入，配合 `yield return` 即可逐行生成，内存占用恒定：

```csharp
using MiniExcelLibs;
using Microsoft.EntityFrameworkCore;

namespace DotNetModulith.Modules.Reports.Application;

internal sealed class MiniExcelExcelBuilder : IExcelBuilder
{
    private const int BatchSize = 5000;

    private readonly IBookExportQueryService _bookQueryService;
    private readonly ReportsDbContext _dbContext;

    public MiniExcelExcelBuilder(IBookExportQueryService bookQueryService, ReportsDbContext dbContext)
    {
        _bookQueryService = bookQueryService;
        _dbContext = dbContext;
    }

    public async Task<int> BuildBookListExcelAsync(
        Stream output,
        CreateBookExportInput filter,
        Guid taskId,
        CancellationToken ct)
    {
        var sheetRows = new CursorEnumerable(_bookQueryService, _dbContext, filter, taskId, ct);
        MiniExcel.SaveAs(output, sheetRows, printHeader: true);
        return sheetRows.TotalRows;
    }

    private sealed class CursorEnumerable : IEnumerable<IDictionary<string, object>>
    {
        private readonly IBookExportQueryService _queryService;
        private readonly ReportsDbContext _dbContext;
        private readonly CreateBookExportInput _filter;
        private readonly Guid _taskId;
        private readonly CancellationToken _ct;

        public int TotalRows { get; private set; }

        public CursorEnumerable(
            IBookExportQueryService queryService,
            ReportsDbContext dbContext,
            CreateBookExportInput filter,
            Guid taskId,
            CancellationToken ct)
        {
            _queryService = queryService;
            _dbContext = dbContext;
            _filter = filter;
            _taskId = taskId;
            _ct = ct;
        }

        public IEnumerator<IDictionary<string, object>> GetEnumerator()
        {
            Guid? lastId = null;
            TotalRows = 0;

            while (true)
            {
                _ct.ThrowIfCancellationRequested();

                var batch = _queryService.FetchBatchAsync(
                    _filter.Keyword, _filter.CategoryId, lastId, BatchSize, _ct)
                    .GetAwaiter().GetResult();

                if (batch.Count == 0)
                    break;

                foreach (var row in batch)
                {
                    yield return new Dictionary<string, object>
                    {
                        ["ISBN"] = row.Isbn,
                        ["书名"] = row.Title,
                        ["作者"] = row.Author,
                        ["出版社"] = row.Publisher,
                        ["出版日期"] = row.PublishDate.ToString("yyyy-MM-dd"),
                        ["分类"] = row.CategoryName,
                        ["总册数"] = row.TotalCopies,
                        ["可借册数"] = row.AvailableCopies,
                        ["状态"] = row.Status
                    };
                    TotalRows++;
                }

                lastId = batch[^1].Id;

                var task = _dbContext.ExportTasks.Find(_taskId);
                if (task is not null)
                {
                    task.UpdateProgress(TotalRows, DateTimeOffset.UtcNow);
                    _dbContext.SaveChanges();
                }
            }
        }

        System.Collections.IEnumerator System.Collections.IEnumerable.GetEnumerator() => GetEnumerator();
    }
}
```

关键设计点：

1. **游标分批**：每批 `BatchSize=5000` 行，使用 `where id > lastId limit batchSize` 游标推进，避免一次性加载全量数据
2. **流式写入**：MiniExcel `SaveAs(Stream, IEnumerable)` 接收 `yield return` 的枚举器，逐行写入输出流，内存占用恒定
3. **进度更新**：每批处理完成后调用 `task.UpdateProgress` 落库，前端轮询可获取实时进度
4. **取消支持**：每批前检查 `CancellationToken`，支持任务取消

### 6.3 MiniExcel 与 ClosedXML 的选型对比

| 维度                | MiniExcel                | ClosedXML                     |
| ------------------- | ------------------------ | ----------------------------- |

```csharp
public async Task<List<BookExportRow>> FetchBatchAsync(
    string? keyword, Guid? categoryId, Guid? lastId, int batchSize, CancellationToken ct)
{
    var query = _dbContext.Books
        .Include(x => x.Category)
        .AsNoTracking()
        .OrderBy(x => x.Id)
        .Take(batchSize);

    if (lastId.HasValue)
        query = query.Where(x => x.Id > lastId.Value);

    if (!string.IsNullOrWhiteSpace(keyword))
    {
        var kw = keyword.Trim();
        query = query.Where(x => x.Title.Contains(kw) || x.Isbn.Contains(kw) || x.Author.Contains(kw));
    }

    if (categoryId.HasValue)
        query = query.Where(x => x.CategoryId == categoryId.Value);

    var entities = await query.ToListAsync(ct);

    return entities.Select(entity => new BookExportRow(
        entity.Isbn, entity.Title, entity.Author, entity.Publisher,
        entity.PublishDate,
        entity.Category?.Name ?? string.Empty,
        entity.TotalCopies, entity.AvailableCopies,
        entity.Status.ToString())).ToList();
}

private IQueryable<BookEntity> BuildQuery(string? keyword, Guid? categoryId)
{
    var query = _dbContext.Books.AsNoTracking().AsQueryable();

    if (!string.IsNullOrWhiteSpace(keyword))
    {
        var kw = keyword.Trim();
        query = query.Where(x => x.Title.Contains(kw) || x.Isbn.Contains(kw) || x.Author.Contains(kw));
    }

    if (categoryId.HasValue)
        query = query.Where(x => x.CategoryId == categoryId.Value);

    return query;
}
```

遵循 [efcore-patterns](file:///d:/Code/dotnet-modulith/.trae/skills/efcore-patterns/SKILL.md) 技能的 `AsNoTracking` 与查询分离建议。`OrderBy(x => x.Id)` 是游标分页的必要条件，保证游标推进的确定性。

## 十、安全与可靠性

### 10.1 权限控制

- 提交、查询、下载均需 `reports.export` 权限
- 查询与下载接口校验 `task.RequestedUserId == currentUserId`，防止越权访问他人产物

### 10.2 幂等与重试

- CAP 内置重试机制，订阅者通过 `task.Status == Pending` 做幂等保护（对应时序图步骤 7 抢占式锁）
- 使用 `UPDATE ... WHERE id = ? AND status = Pending` 的乐观锁方式推进 `Running` 状态，防止 CAP 并发重试导致重复执行
- 失败时标记 `Failed` 并发布失败事件，避免无限重试
- 建议在 `ReportsModule` 配置 CAP 重试策略，限制最大重试次数

### 10.3 产物生命周期

- `ExpiresAt` 默认 7 天，过期后下载接口拒绝签发 URL
- 后台任务（可选 TickerQ Job）扫描 `ExpiresAt < now` 的 `Succeeded` 任务，调用 `IObjectStorageService` 删除对象并标记 `Expired`
- 参考 [scheduled-jobs.md](file:///d:/Code/dotnet-modulith/docs/scheduled-jobs.md) 的 TickerQ 任务编写规范

### 10.4 资源保护

- 提交时数据量预检：`EstimateRowsAsync` 预估行数，超过 `MaxExportRows`（默认 10 万）直接拒绝，避免无效任务
- 单用户提交频率限制（可选，通过中间件实现）
- Excel 生成内存控制：游标分批拉取 + 临时文件流式写入，内存占用恒定（不随数据量增长）
- 临时文件清理：`finally` 块保证临时文件被删除，避免磁盘泄漏

### 10.5 可观测性

- 每个状态转换打 OpenTelemetry Span，关联 `exportTaskId`
- 记录指标：导出请求次数、生成耗时、成功率、平均行数、平均产物大小
- 失败时保留 `LastError` 与堆栈日志，便于排查

```vue
  downloadingId.value = row.exportTaskId
  try {
    const { downloadUrl } = await getExportDownloadUrl(row.exportTaskId)
    const link = document.createElement('a')
    link.href = downloadUrl
    link.download = `图书列表导出_${row.exportTaskId}.xlsx`
    document.body.appendChild(link)
    link.click()
    document.body.removeChild(link)
  } catch (e: any) {
    message.error(e?.message || '下载失败')
  } finally {
    downloadingId.value = null
  }
}

let pollTimer: ReturnType<typeof setInterval> | null = null

function startPolling() {
  pollTimer = setInterval(() => {
    const hasActive = tasks.value.some(t => t.status === 'Pending' || t.status === 'Running')
    if (hasActive) {
      fetchTasks()
    }
  }, 5000)
}

function stopPolling() {
  if (pollTimer) {
    clearInterval(pollTimer)
    pollTimer = null
  }
}

onMounted(() => {
  fetchTasks()
  startPolling()
})

onUnmounted(() => {
  stopPolling()
})
</script>
```

### 11.6 前端注意事项

- **不要长连接等待**：导出是异步任务，前端不阻塞等待完成，通过列表页轮询或通知中心感知
- **轮询策略**：仅当列表中存在 `Pending`/`Running` 任务时才轮询，频率 5 秒，页面销毁时清理定时器
- **下载 URL 时效**：预签名 URL 默认 15 分钟过期，过期后重新点击"下载"按钮申请新 URL
- **过期产物**：`Expired` 状态或 `expiresAt < now` 的任务不展示"下载"按钮，引导用户重新提交
- **失败处理**：`Failed` 状态展示错误提示，引导用户重新提交，不自动重试以免雪崩
- **权限控制**：菜单项与路由均校验 `reports.export` 权限，无权限用户不可见
- **文件名**：下载时通过 `<a download>` 指定文件名，避免预签名 URL 中的随机对象名
